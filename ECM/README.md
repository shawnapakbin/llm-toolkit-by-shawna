# ECM Tool — Extended Context Memory (v2.1.0)

Enables effective 1 million token context windows for LLM agents without exceeding hardware limits. Acts as an external memory store: the agent stores conversation turns, tool outputs, documents, and reasoning traces as retrievable segments with vector embeddings, then queries for the most relevant context before each reasoning turn.

## How It Works

The model's physical context window stays within its hardware limit. ECM provides an unbounded external memory that the agent queries before each turn — retrieving only the highest-scoring segments that fit within a caller-specified token budget. Older segments can be compressed via extractive summarization to keep storage lean.

**Retrieval scoring** (composite):
```
score = (cosine_similarity × 0.7) + (importance × 0.2) + (recency × 0.1)
```
where `recency = 1 / (1 + age_in_hours)` — decays over time.

## MCP Operations

| Action | Description |
|--------|-------------|
| `store_segment` | Persist a memory segment with content, type, and optional importance/metadata |
| `retrieve_context` | Semantic search — returns top-K segments ranked by composite score within a token budget |
| `list_segments` | Paginated list of segments for a session |
| `delete_segment` | Remove a segment by UUID |
| `clear_session` | Delete all segments for a session |
| `summarize_session` | Compress oldest segments into a single summary (extractive, no LLM call) |

## Endpoints

- `GET /health` — `{ ok: true, service: "lm-studio-ecm-tool", version: "2.1.0" }`
- `GET /tool-schema` — Tool schema JSON
- `POST /tools/ecm` — All operations via `action` field
- MCP: `node dist/mcp-server.js` (stdio)

**Default port**: `3342`

## Segment Types

| Type | Use for |
|------|---------|
| `conversation_turn` | User/assistant messages |
| `tool_output` | Results from tool calls |
| `document` | Ingested reference documents |
| `reasoning` | Agent reasoning traces |
| `summary` | Auto-generated by `summarize_session` |

## Tool Schema

```typescript
// store_segment
{
  action: "store_segment",
  sessionId: "my-session-uuid",
  type: "conversation_turn",
  content: "User asked about the deployment pipeline.",
  importance: 0.7,          // optional, 0–1, default 0.5
  metadata: { source: "chat" }  // optional
}

// retrieve_context
{
  action: "retrieve_context",
  sessionId: "my-session-uuid",
  query: "deployment pipeline status",
  topK: 10,         // optional, default 10
  maxTokens: 2048,  // optional, default 4096 — token budget
  minScore: 0.3     // optional — filter by minimum composite score
}

// list_segments
{ action: "list_segments", sessionId: "my-session-uuid", limit: 20, offset: 0 }

// delete_segment
{ action: "delete_segment", sessionId: "my-session-uuid", segmentId: "uuid-here" }

// clear_session
{ action: "clear_session", sessionId: "my-session-uuid" }

// summarize_session
{
  action: "summarize_session",
  sessionId: "my-session-uuid",
  keepNewest: 10   // optional — keep this many newest segments untouched
}
```

## Typical Agent Loop

```typescript
// Before each reasoning turn:
const ctx = await ecm({ action: "retrieve_context", sessionId, query: currentTask, maxTokens: 3000 });
// Inject ctx.data.segments into the prompt

// After each turn, store the result:
await ecm({ action: "store_segment", sessionId, type: "conversation_turn", content: assistantReply });

// Periodically compress when session grows large:
await ecm({ action: "summarize_session", sessionId, keepNewest: 20 });
```

## Token Budget Enforcement

`retrieve_context` fills segments greedily (highest score first) until adding the next segment would exceed `maxTokens`. The response includes:
- `totalTokens` — sum of token counts for returned segments
- `truncated: true` — if additional scored segments were excluded due to budget

Token count is estimated as `Math.ceil(content.length / 4)`.

## Session Isolation

All operations are scoped to `sessionId`. Segments from different sessions never mix. Use UUID v4 session IDs in multi-agent deployments to prevent cross-session leakage.

## Environment Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `ECM_DB_PATH` | `../ecm.db` (relative to script dir) | SQLite database path |
| `ECM_EMBEDDINGS_MODE` | `lmstudio` | `lmstudio` or `mock` (for testing) |
| `ECM_EMBEDDING_MODEL` | `nomic-ai/nomic-embed-text-v1.5` | LM Studio embedding model |
| `PORT` | `3342` | HTTP server port |

## LM Studio Integration

```json
{
  "mcpServers": {
    "ecm": {
      "command": "node",
      "args": ["ECM/dist/mcp-server.js"],
      "env": {
        "ECM_DB_PATH": "./ecm.db",
        "ECM_EMBEDDINGS_MODE": "lmstudio",
        "ECM_EMBEDDING_MODEL": "nomic-ai/nomic-embed-text-v1.5"
      }
    }
  }
}
```

Requires LM Studio running with an embedding model loaded (e.g. `nomic-ai/nomic-embed-text-v1.5`).
